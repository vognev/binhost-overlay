# Маніфест Бутстрапу Чистокровного LLVM/Clang Сисрута (Gentoo Crossdev)

Коли хостовий компілятор налаштований на **GNU-стек** (`libgcc_s`, `libstdc++`), стандартна автоматика `crossdev` пасує перед LLVM-таргетом, оскільки намагається застосувати лінійну логіку для мультитаргетного компілятора. 

**Головний інженерний фінт:** Повністю ігноруємо крос-GCC і змушуємо хостовий Clang збирати рантайми *в інтересах таргета* через створення ізольованої віртуальної категорії в крос-оверлеї. Це дозволяє використовувати мультитаргетну природу Clang на повну.

---

## Етап 1. Створення ізольованого LLVM крос-оверлею

Замість того, щоб битися з глобальними масками хоста, підсовуємо Portage дзеркало категорій прямо в локальний репозиторій `crossdev`. Це змусить ебілди розуміти, що вони збираються як крос-пакети, чиї артефакти мають летіти строго в сисрут, але силами хоста.

```bash
cd /var/db/repos/crossdev
echo "cross_llvm-x86_64-pc-linux-gnu" >> profiles/categories
mkdir cross_llvm-x86_64-pc-linux-gnu && cd cross_llvm-x86_64-pc-linux-gnu

# Лінкаємо ебілди рантаймів та враперів з основного дерева Gentoo
ln -sf /var/db/repos/gentoo/llvm-runtimes/{compiler-rt,libunwind,libcxxabi,libcxx} .
ln -sf /var/db/repos/gentoo/sys-devel/clang-crossdev-wrappers .
```

## Етап 2. Розкручування рантаймів (Strict Order)

У чистокровному LLVM-стеку порядок збірки має фундаментальне значення, де кожен наступний крок спирається на об'єктники попереднього. Порушення порядку призведе до неможливості лінкування.

1. compiler-rt (builtins) — Фундамент компілятора

Збирається першим. Він не вимагає жодних сторонніх ліб і закидає в систему базові низькорівневі об'єктники компілятора та вбудовані функції (libclang_rt.builtins.a, clang_rt.crtbegin.o). Вони лягають прямо в системну директорію Clang хоста під архітектурним префіксом таргета:
Bash

`emerge -1av cross_llvm-x86_64-pc-linux-gnu/compiler-rt`

2. libunwind — Статичний механізм розкручування

Форсуємо збірку строго в статику (static-libs). Статична ліба не потребує динамічного лінкування з іншими рантаймами під час власної збірки, але стає опорною точкою, яка розриває циклічну помилку лінкера unable to find library -lunwind для майбутніх shared-ліб.
Bash

`USE="static-libs" emerge -1av cross_llvm-x86_64-pc-linux-gnu/libunwind`

3. libcxxabi — Низькорівневий шар мови C++

Забезпечує роботу виключень (throw/catch), приведення типів та потоків. Обов'язково забороняємо використання бінарних пакетів хоста (--usepkg=n), щоб архітектурні оптимізації Skylake запікалися безпосередньо в об'єктники.
Bash

`emerge -1av --usepkg=n cross_llvm-x86_64-pc-linux-gnu/libcxxabi`

4. libcxx — Високорівнева стандартна бібліотека C++

Фінальний акорд розгортання C++ рантайму. Розгортає саму libc++.so та весь набір стандартних заголовків C++ у сисрут (/usr/x86_64-pc-linux-gnu/usr/lib64/ та /usr/include/c++/v1/).
Bash

`emerge -1av --usepkg=n cross_llvm-x86_64-pc-linux-gnu/libcxx`

Етап 3. Декларація незалежності Крос-Конфігу

Оскільки хостовий Clang є «гнутим», крос-конфіг компілятора обов'язково повинен містити явні вказівки на LLVM-рантайми. Інакше через глобальний @../gentoo-runtimes.cfg він буде постійно скочуватися в хостові GNU-дефолти й шукати GCC-ліби, ламаючи крос-компіляцію.

Вміст файлу /etc/clang/cross/x86_64-pc-linux-gnu.cfg:

```bash
--target=x86_64-pc-linux-gnu
--sysroot=/usr/x86_64-pc-linux-gnu
@../gentoo-runtimes.cfg
-rtlib=compiler-rt
-unwindlib=libunwind
-stdlib=libc++
-fuse-ld=lld
```

Примітка: Цей файл crossdev ніколи не затре самостійно, тому конфіг повністю захищений від оновлень тулчейну.
Результат

Після виконання цієї послідовності крос-тулчейн стає автономним. Будь-які базові системні пакети починають проходити тести лінкера без залучення GNU-компонентів:
Bash

x86_64-pc-linux-gnu-emerge -1av sys-libs/zlib
Результат: sys-libs/zlib-1.3.2-r1 merged успішно!
